# 문서 작성 가이드

**원칙**: 복잡하지 않게, 명확하게, 실용적으로

---

## 기본 규칙

### 1. 길이보다 복잡도 관리
- 길이는 고정 제한하지 않는다.
- 문제는 문서의 분량이 아니라, 한 번에 이해하기 어려운 복잡도다.
- 문서가 길어도 괜찮지만, 섹션별 역할과 메시지는 단순해야 한다.

### 2. 구조
```
✅ 필수:
- 핵심 규칙
- 최소 예시
- 대표 안티 패턴

❌ 불필요:
- 장황한 배경 설명
- 중복 예시
- 과도한 체크리스트
- 한 섹션에 여러 주제 혼합
```

### 3. 작성 순서
1. 규칙을 한 문장으로 정의
2. 최소 예시 추가
3. 오해를 부르는 안티 패턴 1개 추가
4. 필요 시 예외 조건을 짧게 명시

---

## 안티 패턴

### ❌ 과도한 설명
```markdown
## 왜 이 규칙이 필요한가?

역사적으로 살펴보면, 1990년대부터...
소프트웨어 공학의 관점에서...
학술적 연구에 따르면...

(3페이지 계속...)
```

**해결책**: 삭제. 바로 규칙만 적기.

### ❌ 중복된 예시
```markdown
## 예시

### 예시 1: 컴포넌트
...

### 예시 2: 컴포넌트 (거의 같음)
...

### 예시 3: 컴포넌트 (또 같음)
...
```

**해결책**: 서로 다른 결정을 설명하는 예시만 남기기.

### ❌ 과도한 체크리스트
```markdown
## 체크리스트

작성 전:
- [ ] ...
- [ ] ...
(20개 항목)

작성 중:
- [ ] ...
(20개 항목)

작성 후:
- [ ] ...
(20개 항목)
```

**해결책**: 핵심 3-5개만.

---

## 작성 후 검증

**질문**:
- [ ] 각 섹션이 하나의 주제만 다루는가?
- [ ] 중복 예시/중복 설명이 제거됐는가?
- [ ] 읽는 사람이 바로 실행할 수 있는 규칙인가?
- [ ] 예외가 있다면 기준이 명확한가?

---

## 좋은 예시

```markdown
# 짧은 제목

## 규칙
- 규칙 1
- 규칙 2

## 예시
[코드 1개]

## 안티 패턴
❌ 나쁜 예
✅ 좋은 예

끝.
```

**핵심**: 길이보다 구조와 명확성이 우선

---

**요약**: 제한할 것은 줄 수가 아니라 복잡도
